    05. Split Routing¶

    Что такое Split Routing¶

    Split Routing, или Split Tunnel, — это режим работы VPN, при котором через VPN-туннель направляется только определённый трафик, а остальной продолжает использовать обычное интернет-соединение.

    В отличие от Full Tunnel, где весь сетевой трафик проходит через VPN, Split Routing позволяет выбирать, какие ресурсы должны использовать защищённый канал.

    Это уменьшает задержки, снижает нагрузку на VPS и позволяет одновременно пользоваться локальными ресурсами домашней сети и VPN.

    Full Tunnel и Split Tunnel¶

    Full Tunnel¶

                    Интернет
                    │
                    ▼
                 VPS/VPN
                    ▲
                    │
          Весь сетевой трафик
                    ▲
                    │
                 Устройство

    Преимущества:

    • весь трафик проходит через VPN;
    • используется единый внешний IP;
    • не требуется поддерживать отдельные списки маршрутов.

    Недостатки:

    • увеличивается задержка;
    • возрастает нагрузка на VPS;
    • весь интернет зависит от работоспособности VPN;
    • локальные ресурсы могут потребовать дополнительной настройки;
    • через VPS проходит даже тот трафик, которому VPN не требуется.

    Split Tunnel¶

                        Интернет
                 ▲            ▲
                 │            │
          VPN-трафик      Обычный трафик
                 │            │
                 ▼            ▼
              VPS         Провайдер
                 ▲            ▲
                  \          /
                   \        /
                    Устройство

    Преимущества:

    • VPN используется только там, где это необходимо;
    • основной интернет работает напрямую;
    • уменьшается задержка для обычного трафика;
    • снижается нагрузка на VPS;
    • домашняя сеть остаётся доступной;
    • можно отдельно управлять маршрутами для разных сервисов.

    Именно такой подход используется в данном проекте.

    Где принимается решение¶

    Важно понимать принцип работы архитектуры.

    Сервер AmneziaWG не принимает решение, какой трафик должен идти через VPN.

    Сервер выполняет только обработку пакетов, которые уже были направлены в туннель клиентом.

    Вся логика Split Routing находится на стороне клиентских устройств.

    В данном проекте используются два разных подхода:

    • Windows-клиенты используют маршруты из параметра AllowedIPs;
    • роутер Cudy использует Podkop, nftables, Policy Routing, TPROXY и sing-box.

    Архитектура проекта¶

                        Интернет
                        ▲
                        │
                 NAT / Masquerade
                        │
                        ▼
                  VPS / AmneziaWG
                        ▲
                        │
                     awg0
                        ▲
                        │
                 ┌──────────────┐
                 │              │
             Windows         Cudy OpenWrt
                 │              │
             AllowedIPs    Podkop + sing-box
                                │
                         Policy Routing

    Серверная часть для обоих вариантов остаётся одинаковой.

    Различается только способ, которым клиент определяет трафик, направляемый в VPN.

    Split Routing на Windows¶

    На Windows используется классический механизм маршрутизации.

    После подключения клиента создаются дополнительные маршруты через VPN-интерфейс.

    При этом основной шлюз системы остаётся домашним маршрутизатором.

    Получается следующая схема:

    Локальная сеть
        │
        ▼
Домашний роутер
        │
        ├──────────────► Обычный интернет
        │
        └──────────────► Маршруты из AllowedIPs
                          │
                          ▼
                     AmneziaWG
                          │
                          ▼
                         VPS

    Это означает:

    • обычные сайты открываются напрямую;
    • только указанные IP-адреса и подсети используют VPN;
    • локальная сеть продолжает работать через обычный шлюз;
    • выбор маршрута выполняет операционная система Windows.

    Основным параметром является:

    AllowedIPs = NETWORK_1, NETWORK_2, NETWORK_3

    Именно он определяет, какие сети будут доступны через VPN-интерфейс.

    AllowedIPs — это не список доменов

    В параметр AllowedIPs нельзя добавить:

    youtube.com
discord.com
chatgpt.com

    AmneziaWG работает с IP-адресами и CIDR-подсетями:

    203.0.113.10/32
198.51.100.0/24

    Split Routing на Cudy OpenWrt¶

    Для маршрутизатора используется более гибкая схема.

    Вместо обычного списка маршрутов применяются несколько компонентов:

    • Podkop;
    • nftables;
    • Policy Routing;
    • TPROXY;
    • sing-box;
    • AmneziaWG.

    Такая архитектура позволяет принимать решение не только по заранее заданным IP-адресам, но и по доменным правилам и динамически разрешённым адресам.

    Как работает маршрутизация на OpenWrt¶

    Последовательность обработки выглядит следующим образом:

    Устройство
      │
      ▼
Cudy OpenWrt
      │
      ▼
Podkop определяет,
нужно ли использовать VPN
      │
      ▼
nftables устанавливает fwmark
      │
      ▼
ip rule
      │
      ▼
таблица podkop
      │
      ▼
local default dev lo
      │
      ▼
TPROXY
      │
      ▼
sing-box
      │
      ▼
AmneziaWG / awg0
      │
      ▼
VPS
      │
      ▼
NAT / Masquerade
      │
      ▼
Интернет

    Именно эта архитектура используется в данной системе.

    Почему используется sing-box¶

    Многие VPN-решения работают исключительно через изменение таблицы маршрутизации.

    В данном проекте используется более гибкий подход.

    После определения нужного трафика Podkop помечает соединение средствами nftables.

    Затем Policy Routing направляет отмеченный трафик в локальную таблицу podkop, откуда он попадает в TPROXY и передаётся локальному экземпляру sing-box.

    После этого sing-box отправляет соединение через интерфейс AmneziaWG.

    Такой подход позволяет:

    • работать с доменными правилами;
    • использовать динамически разрешаемые IP-адреса;
    • применять крупные и обновляемые списки;
    • не изменять основной маршрут системы;
    • не отправлять через VPN весь домашний трафик;
    • централизованно управлять маршрутизацией домашней сети.

    Роль VPS¶

    Сервер не знает, какой именно трафик должен использовать VPN.

    Он выполняет только следующие задачи:

    • принимает соединения AmneziaWG;
    • расшифровывает пакеты;
    • пересылает их в Интернет;
    • выполняет NAT или Masquerade;
    • принимает ответный трафик;
    • возвращает ответы клиенту.

    Таким образом, логика выбора трафика находится исключительно на стороне клиентов.

    Преимущества такой архитектуры¶

    Используемая схема имеет несколько преимуществ:

    • основной интернет продолжает работать напрямую;
    • VPN используется только там, где это необходимо;
    • отсутствует лишняя нагрузка на VPS;
    • домашняя сеть остаётся доступной;
    • удобно обновлять списки маршрутизации;
    • можно использовать разные правила для разных клиентов;
    • систему можно масштабировать без изменения серверной части.

    Откуда брать IP-адреса для Split Routing¶

    Для классического Split Routing через AllowedIPs необходимо сформировать список IP-адресов и подсетей.

    Добавлять их вручную неудобно, особенно для крупных сервисов, использующих большое количество адресов, автономных систем и CDN.

    В данном проекте для подготовки исходного списка использовался сервис:

    IP Address Collection and Management Service

    Сервис позволяет:

    • выбрать необходимые порталы и группы;
    • исключить ненужные ресурсы;
    • сформировать список IP-адресов и подсетей;
    • выбрать формат выгрузки;
    • сохранить результат в файл;
    • использовать полученные сети при создании AllowedIPs.

    Назначение сервиса

    Сервис используется как внешний источник данных для формирования маршрутов.

    Он не является частью AmneziaWG, Windows-клиента, OpenWrt или данного репозитория.

    Базовый порядок работы¶

    1. Откройте страницу:

    https://iplist.opencck.org/ru/

    1. Выберите необходимые порталы или группы.

    2. При необходимости добавьте исключения.

    3. Выберите подходящий формат выгрузки.

    4. Сформируйте список.

    5. Сохраните результат в файл.

    6. Проверьте содержимое файла.

    7. Удалите пустые строки и дубликаты.

    8. Объедините сети в строку через запятую.

    9. Вставьте полученную строку в AllowedIPs.

    Подготовка файла в Linux или WSL¶

    Предположим, полученный файл называется:

    networks.txt

    Проверить первые строки:

    head -n 20 networks.txt

    Удалить пустые строки:

    sed '/^[[:space:]]*$/d' networks.txt > networks-clean.txt

    Удалить повторяющиеся записи:

    sort -u networks-clean.txt > networks-unique.txt

    Проверить итоговое количество сетей:

    wc -l networks-unique.txt

    Показать первые записи:

    head -n 20 networks-unique.txt

    Формирование строки AllowedIPs¶

    Исходный файл может выглядеть так:

    203.0.113.0/24
198.51.100.10/32
192.0.2.0/24

    Для конфигурации AmneziaWG сети необходимо объединить в одну строку через запятую:

    paste -sd, networks-unique.txt | sed 's/,/, /g' > allowedips.txt

    Проверить результат:

    batcat allowedips.txt

    Результат:

    203.0.113.0/24, 198.51.100.10/32, 192.0.2.0/24

    Полученную строку нужно вставить в клиентскую конфигурацию:

    [Peer]
PublicKey = SERVER_PUBLIC_KEY
PresharedKey = CLIENT_PRESHARED_KEY
Endpoint = SERVER_IP:SERVER_PORT
AllowedIPs = 203.0.113.0/24, 198.51.100.10/32, 192.0.2.0/24
PersistentKeepalive = 25

    Документационные адреса

    Подсети:

    192.0.2.0/24
198.51.100.0/24
203.0.113.0/24

    приведены только как примеры.

    Не копируйте их в рабочую конфигурацию.

    Почему списки необходимо обновлять¶

    Списки IP-адресов не являются постоянными.

    Сервисы могут:

    • менять хостинг;
    • использовать новые подсети;
    • переносить ресурсы между дата-центрами;
    • подключать или отключать CDN;
    • использовать разные адреса в зависимости от региона;
    • размещать несколько независимых сервисов в одной подсети.

    Поэтому список, сформированный однажды, со временем может стать неполным или содержать устаревшие записи.

    Периодически обновляйте данные

    Сохраняйте исходные параметры генерации списка.

    Это позволит позднее повторно сформировать актуальный файл и обновить AllowedIPs.

    Общие CDN и широкие подсети¶

    Один IP-адрес или одна подсеть может использоваться сразу несколькими независимыми ресурсами.

    Если добавить в AllowedIPs слишком широкий диапазон, через VPN может пойти больше трафика, чем планировалось.

    Например, добавление крупной подсети CDN может затронуть:

    • нужный сервис;
    • сторонние сайты;
    • обновления приложений;
    • изображения и видеоматериалы;
    • рекламные и аналитические домены;
    • другие ресурсы, размещённые у того же провайдера.

    Поэтому рекомендуется:

    • выбирать только необходимые группы;
    • не добавлять слишком широкие диапазоны без проверки;
    • сравнивать маршруты до и после изменения;
    • хранить исходный список отдельно;
    • тестировать работу обычных сайтов после обновления.

    AllowedIPs и Podkop решают разные задачи¶

    Классический Split Routing через AllowedIPs хорошо подходит для:

    • Windows;
    • iPhone;
    • отдельных клиентских устройств;
    • конфигураций без дополнительного маршрутизатора;
    • статических списков IP-адресов.

    Но у этого способа есть ограничение: он работает с IP-адресами, а не с доменными именами.

    Podkop на OpenWrt решает задачу иначе.

    Он позволяет использовать более гибкие правила и передавать выбранный трафик через sing-box и AmneziaWG.

    Поэтому в данном проекте используются оба подхода:

    Клиент Механизм выбора трафика
    Windows AllowedIPs
    iPhone AllowedIPs
    Cudy OpenWrt Podkop, nftables, Policy Routing, TPROXY и sing-box

    Проверка маршрутов после обновления списка¶

    После добавления нового списка необходимо проверить, что маршрутизация работает ожидаемым образом.

    На Windows:

    route print

    Для проверки конкретного адреса:

    tracert TARGET_IP

    На Linux или WSL:

    ip route get TARGET_IP

    Ожидаемое поведение:

    • адрес из AllowedIPs направляется через VPN-интерфейс;
    • обычный адрес использует основной шлюз;
    • локальная сеть остаётся доступной;
    • внешний IP меняется только для выбранного трафика.

    Не используйте 0.0.0.0/0 для Split Routing

    Значение:

    AllowedIPs = 0.0.0.0/0

    отправляет через туннель весь IPv4-трафик.

    Это режим Full Tunnel, а не Split Routing.
