# 04. Firewall

## 🎯 Цель

После установки AmneziaWG сервер уже имеет настроенный брандмауэр. В отличие от большинства VPN-руководств, здесь не требуется вручную создавать десятки правил `iptables` или `nftables` — установщик делает это автоматически.

В этой главе мы подробно разберём:

- какой брандмауэр используется;
- какие правила создаёт AmneziaWG;
- как проверить их работу;
- как безопасно диагностировать проблемы;
- какие изменения можно вносить, а какие лучше не трогать.

---

# 📋 Что используется в данной конфигурации

На момент написания документации сервер имеет следующую конфигурацию:

- **Ubuntu 24.04 LTS**
- **AmneziaWG Server**
- **nftables**
- **UFW установлен, но отключён**

Проверить состояние UFW можно командой:

```bash
sudo ufw status verbose
```

Результат:

```text
Status: inactive
```

Это означает, что UFW не участвует в фильтрации трафика.

---

# Почему используется nftables

Современные версии Ubuntu используют **nftables** как основной механизм фильтрации сетевого трафика.

Во время установки AmneziaWG создаёт собственную таблицу правил и самостоятельно управляет ей.

Проверить это можно командой:

```bash
sudo nft list ruleset
```

В результате будет отображена таблица, похожая на эту:

```text
table ip awg-awg0
```

Именно эта таблица отвечает за работу VPN.

---

# 🔒 Автоматическая настройка

После установки AmneziaWG автоматически:

- открывает UDP-порт VPN;
- создаёт правила маршрутизации;
- включает NAT (маскарадинг);
- настраивает пересылку пакетов;
- ограничивает доступ к веб-панели;
- добавляет оптимизацию MSS для предотвращения проблем с MTU.

Другими словами, пользователю **не требуется вручную писать правила nftables** для стандартной работы сервера.

---

# 🛡️ Разбор правил nftables

Ниже приведён разбор правил, автоматически созданных AmneziaWG.

## Цепочка `input`

```text
chain input {
    type filter hook input priority filter; policy accept;

    udp dport 53925 accept

    iifname "awg0" ip saddr { 10.66.66.2, 10.66.66.3 } tcp dport 8080 accept

    iifname "awg0" tcp dport 8080 drop
}
```

Эта цепочка отвечает за входящие подключения к серверу.

### Разрешение подключения к VPN

```text
udp dport 53925 accept
```

Это правило открывает UDP-порт, на котором работает сервер AmneziaWG.

Проверить, какой порт используется, можно командой:

```bash
sudo ss -lun
```

или

```bash
sudo ss -tulpen
```

В нашей конфигурации используется:

```text
UDP 53925
```

Если изменить порт VPN через веб-панель, соответствующее правило будет автоматически обновлено.

---

### Доступ к веб-панели

Следующее правило выглядит значительно интереснее:

```text
iifname "awg0" ip saddr { 10.66.66.2, 10.66.66.3 } tcp dport 8080 accept
```

Разберём его по частям.

| Элемент | Значение |
|---------|----------|
| `iifname "awg0"` | пакет пришёл через VPN-интерфейс |
| `ip saddr` | IP-адрес клиента внутри VPN |
| `tcp dport 8080` | обращение к веб-панели |

Это означает, что доступ к веб-интерфейсу получают **только авторизованные VPN-клиенты**.

В нашей конфигурации разрешены:

```text
10.66.66.2
10.66.66.3
```

При добавлении нового клиента AmneziaWG автоматически обновляет это правило.

---

### Защита панели управления

Следующее правило:

```text
iifname "awg0" tcp dport 8080 drop
```

запрещает доступ к панели всем VPN-клиентам, которых нет в списке разрешённых адресов.

Иными словами:

- клиент существует внутри VPN;
- но если его IP отсутствует в списке разрешённых;
- подключение к панели будет немедленно отклонено.

Это дополнительный уровень защиты административного интерфейса.

---

## Почему панель не видна из Интернета

Проверим, какие адреса прослушивает веб-панель:

```bash
sudo ss -tulpen
```

Результат:

```text
10.66.66.1:8080
```

Обрати внимание — здесь **нет**:

```text
0.0.0.0:8080
```

Это означает, что сервис слушает **только VPN-интерфейс**.

Даже если на VPS открыть порт `8080`, подключиться к панели извне всё равно не получится.

Это одна из сильных сторон архитектуры AmneziaWG: административный интерфейс по умолчанию изолирован от Интернета.

# 🔀 Цепочка `forward`

```text
chain forward {
    type filter hook forward priority filter; policy accept;

    oifname "awg0" tcp flags syn / syn,rst tcp option maxseg size set rt mtu

    iifname "awg0" accept

    iifname "ens3" oifname "awg0" ct state established,related accept

    iifname "ens3" oifname "awg0" drop
}
```

Цепочка `forward` отвечает за пересылку пакетов между сетевыми интерфейсами.

Именно благодаря этим правилам VPN-клиенты получают доступ в Интернет через VPS.

---

## Настройка MSS (MTU Fix)

Первое правило:

```text
oifname "awg0" tcp flags syn / syn,rst tcp option maxseg size set rt mtu
```

На первый взгляд выглядит сложно, однако выполняет очень важную задачу.

Оно автоматически изменяет значение **TCP MSS (Maximum Segment Size)** в момент установки соединения.

Это позволяет избежать типичных проблем:

- сайты открываются не полностью;
- некоторые сервисы перестают отвечать;
- зависают HTTPS-соединения;
- появляются ошибки из-за неправильного MTU.

Во многих руководствах подобное правило приходится добавлять вручную.

AmneziaWG создаёт его автоматически.

---

## Разрешение трафика от VPN-клиентов

Следующее правило:

```text
iifname "awg0" accept
```

означает:

> весь трафик, поступающий с VPN-интерфейса `awg0`, разрешён к дальнейшей обработке.

Именно благодаря этому клиенты могут пользоваться Интернетом через VPS.

---

## Разрешение обратного трафика

Следующее правило:

```text
iifname "ens3" oifname "awg0" ct state established,related accept
```

разрешает только ответный трафик.

Это означает:

- клиент отправил запрос;
- сервер получил ответ из Интернета;
- ответ возвращается обратно через VPN.

Использование состояния соединения (`ct state established,related`) является стандартной практикой при настройке современных межсетевых экранов.

---

## Блокировка остальных соединений

Последнее правило:

```text
iifname "ens3" oifname "awg0" drop
```

запрещает любые новые подключения из Интернета напрямую в VPN.

Таким образом:

- ответы на существующие соединения разрешены;
- новые подключения извне — запрещены.

Это значительно повышает безопасность сервера.

---

# 🌍 Цепочка `postrouting`

```text
chain postrouting {
    type nat hook postrouting priority srcnat; policy accept;

    oifname "ens3" masquerade
}
```

После обработки пакета наступает этап NAT.

---

## Что делает `masquerade`

Правило:

```text
oifname "ens3" masquerade
```

подменяет исходный IP-адрес VPN-клиента адресом самого VPS.

Например:

До NAT:

```text
10.66.66.2 → github.com
```

После NAT:

```text
<VPS Public IP> → github.com
```

Ответ приходит обратно на VPS, после чего ядро Linux автоматически возвращает пакет нужному VPN-клиенту.

Без этого правила VPN-клиенты не смогли бы пользоваться Интернетом.

Именно `masquerade` делает Full Tunnel возможным.

---

# 🔄 IP Forwarding

Для пересылки пакетов одного NAT недостаточно.

Необходимо разрешить ядру Linux выполнять маршрутизацию.

Проверить это можно командой:

```bash
sysctl net.ipv4.ip_forward
```

Правильный результат:

```text
net.ipv4.ip_forward = 1
```

Если значение равно `0`, VPN не сможет пересылать трафик между интерфейсами.

Во время установки AmneziaWG этот параметр автоматически включается.

# ⚠️ Чего не стоит делать

Хотя правила `nftables` можно редактировать вручную, делать это без необходимости не рекомендуется.

В частности, не следует:

- изменять таблицу `awg-awg0` вручную;
- удалять правила `masquerade`;
- отключать `IP Forwarding`;
- изменять правила доступа к веб-панели напрямую через `nft`.

AmneziaWG самостоятельно управляет своей таблицей правил. После изменения конфигурации VPN или добавления новых клиентов правила могут быть автоматически обновлены.

Если требуется изменить параметры работы VPN, рекомендуется выполнять это через веб-панель AmneziaWG, а не редактировать правила вручную.
